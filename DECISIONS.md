# Architecture Decisions

These are the locked decisions for the Daisy platform M1 build. They live here so they don't get re-litigated in every wave. Source: `docs/M1-build-plan.md` §3.

## Locked at M1 kick-off

| Decision             | Choice                                                                                                                                                                                                       | Why                                                                                                                                                                                  |
| -------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Repo strategy        | Single flat repo per app, not monorepo. `daisy-platform` holds the portal + Supabase migrations + Edge Functions. `daisy-medical` (M3) is its own repo. No Turborepo, no npm workspaces, no shared packages. | Each Netlify site connects 1:1 to a repo. Drops monorepo machinery we don't need. Shared types/utils duplicate between repos when M3 lands; trivial cost given how little is shared. |
| Build tool           | Vite 6 + React 19 + TypeScript 5                                                                                                                                                                             | PRD §2.1; React 19 stable enough for Wave 1 to ship on.                                                                                                                              |
| Router               | React Router v7                                                                                                                                                                                              | Simpler than TanStack Router, sufficient for M1 scope, broad ecosystem. v7 ships as the single `react-router` package; the old `react-router-dom` split is deprecated.               |
| Server state         | TanStack Query v5                                                                                                                                                                                            | PRD §2.1.                                                                                                                                                                            |
| Client state         | Zustand 5 (persisted)                                                                                                                                                                                        | PRD §2.1.                                                                                                                                                                            |
| Forms                | React Hook Form + Zod                                                                                                                                                                                        | PRD §2.1.                                                                                                                                                                            |
| Charts               | Recharts                                                                                                                                                                                                     | PRD §2.1.                                                                                                                                                                            |
| UI primitives        | shadcn/ui on Radix + Tailwind 4                                                                                                                                                                              | shadcn copies primitives into the repo so we own them; Tailwind 4 with the `@tailwindcss/vite` plugin and CSS-first `@theme` config.                                                 |
| Map provider         | Google Maps JavaScript API + Geocoding API                                                                                                                                                                   | PRD §21.1 — no fallback to Leaflet now that GCP is being set up Day 0.                                                                                                               |
| Booking reference    | Single global `da_bookings_seq` PostgreSQL sequence; reset January 1 each year                                                                                                                               | Simpler than per-franchisee; format `DA-{YYYY}-{number:05d}-{seq}`.                                                                                                                  |
| Money                | Integer pence everywhere                                                                                                                                                                                     | PRD §4.1. `formatPence(p)` is the only money formatter; never use raw numbers in UI.                                                                                                 |
| Time                 | UTC in DB, `Europe/London` for display, `Intl.DateTimeFormat` not raw `toISOString().split('T')[0]`                                                                                                          | Avoids BST off-by-one bugs.                                                                                                                                                          |
| Mutations            | All via Edge Functions (service_role) — clients use anon key + RLS for reads                                                                                                                                 | PRD §4.18.                                                                                                                                                                           |
| Audit log            | `da_activities` insert from every Edge Function mutation; never from client                                                                                                                                  | PRD §4.15.                                                                                                                                                                           |
| Mobile target        | Tablet-first responsive (iPad). HQ desktop is primary.                                                                                                                                                       | Jenni mentioned iPad use; phone is M2+/franchisee priority.                                                                                                                          |
| Encryption (medical) | Defer to M3 — flag here as "to decide before Wave 12"                                                                                                                                                        | Not in M1 scope.                                                                                                                                                                     |
| Email templates      | Defer to M2                                                                                                                                                                                                  | Not in M1 scope.                                                                                                                                                                     |

## Decisions taken in Wave 1A beyond the plan

| Decision                                                                | Choice                                                                                                                                                                                                                   | Why                                                                                                                                                                                                                                                                           |
| ----------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| ESLint config style                                                     | Flat config (`eslint.config.js`)                                                                                                                                                                                         | ESLint 9 default; the legacy `.eslintrc` format is deprecated.                                                                                                                                                                                                                |
| `tsconfig` layout                                                       | Project references with `tsconfig.app.json` (src) and `tsconfig.node.json` (vite/vitest configs)                                                                                                                         | Standard Vite 6 + TS 5 layout; lets `tsc -b` cache build state across the two graphs.                                                                                                                                                                                         |
| shadcn install                                                          | Wrote the primitives directly rather than running `npx shadcn init` interactively                                                                                                                                        | The CLI's interactive setup is awkward inside a non-interactive agent shell, and shadcn's whole point is that you own the source — there's no behavioural difference. The 11 primitives in `src/components/ui/` are byte-comparable to what `shadcn add` would have produced. |
| `RoleContext` table-missing fallback                                    | Treat "table doesn't exist" as `notProvisioned: false` so the placeholder dashboard still renders during local dev before Wave 1B's migrations land                                                                      | Otherwise every Wave 1A smoke test would require Wave 1B to be merged first. Once `da_franchisees` exists, the missing-row case correctly routes to `/unauthorized`.                                                                                                          |
| CI build env                                                            | Set placeholder `VITE_SUPABASE_URL` / `VITE_SUPABASE_ANON_KEY` for the GitHub Actions build step only                                                                                                                    | The Supabase client throws at module load if either is missing; placeholders let the static build complete without exposing real keys to CI. Real keys live in Netlify env, never in CI.                                                                                      |
| `husky`/`lint-staged`/`prettier` deps installed but not configured here | Per coordination: Agent 1C owns `.husky/`, `.prettierrc`, `.lintstagedrc`, `.prettierignore`. We install the tools and add `prepare: husky` to `package.json` so the postinstall hook works once 1C's config files land. |                                                                                                                                                                                                                                                                               |

## Decisions taken after M1 demo prep (2026-04-30)

| Decision                                      | Choice                                                                                                                                            | Why                                                                                                                                                                                                                                                                                                                             |
| --------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Auth method for franchisees**               | **Google SSO is the primary path.** Email + password kept as a fallback for fixture/test accounts. Magic links abandoned.                         | Real franchisees use Google or Workspace email. "Click your Google account" is a much better UX than "wait for a magic link that expires in an hour, click it before someone else does, and hope email deliverability works". Magic links in Supabase's free SMTP are also rate-limited (~2/hour) and frequently spam-foldered. |
| **Auto-link `auth.users` → `da_franchisees`** | DB trigger (migration 012) updates `da_franchisees.auth_user_id` by case-insensitive email match on first auth-user insert (and on email change). | Lets HQ bulk-import franchisee data with `auth_user_id = NULL`, then have the link form automatically the first time the franchisee signs in via Google. RoleContext logic stays unchanged.                                                                                                                                     |
| **Bulk franchisee onboarding**                | One-shot SQL via Supabase Management API with `INSERT … ON CONFLICT (email) DO UPDATE`. No new Edge Function.                                     | Idempotent, re-runnable if the input changes. The existing `create-franchisee` Edge Function is still the right path for one-off HQ onboarding via the UI form; for 60+ rows at once, direct SQL is simpler.                                                                                                                    |

## Locked at M2 kick-off (2026-05-27)

Source: `docs/M2-build-plan.md` §3 + the M2 swarm execution plan. M1 grew past its planned migration count (now at `018`), so M2 migrations start at **`019`**.

| Decision                            | Choice                                                                                                                                                                                                                                                                       | Why                                                                                                                                                                                                                                    |
| ----------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Stripe Connect model**            | **Standard Connect.** Franchisees own their own Stripe dashboards (refunds, payouts, disputes). Daisy HQ takes a 2% `application_fee_amount`.                                                                                                                                | Custom Connect would put dispute liability on the platform — out of scope.                                                                                                                                                             |
| **Connect onboarding mechanism**    | **Account Links (Connect Onboarding), NOT OAuth.** Server creates the connected account via the Accounts API, generates an `account_onboarding` Account Link, redirects the franchisee, then syncs `stripe_connected` via the `account.updated` webhook + `charges_enabled`. | Stripe flags OAuth as "not recommended for new Connect platforms" (verified May 2026). Account Links also removes the `STRIPE_CONNECT_CLIENT_ID` and redirect-URI allowlist — less platform setup. See `docs/stripe-connect-setup.md`. |
| **Private-booking payment surface** | **Stripe Payment Links** created on the connected account with the application fee. URL stored on `da_course_instances.stripe_payment_link`.                                                                                                                                 | PRD §13.3. Public Checkout Sessions are M3.                                                                                                                                                                                            |
| **Booking row creation timing**     | Booking row created on the **`checkout.session.completed` webhook**, never on Payment Link generation.                                                                                                                                                                       | No half-paid orphan rows (PRD §5.4).                                                                                                                                                                                                   |
| **Spot decrement**                  | Inside the `stripe-webhook` Edge Function via `decrement_spots()`. On `false` (overbook), flag booking `payment_status='manual'` + activity `metadata={overbooking:true}`.                                                                                                   | Atomic with the booking insert.                                                                                                                                                                                                        |
| **Calendar UI**                     | Custom `<MonthCalendar>` (Tailwind grid) in `src/components/daisy/`.                                                                                                                                                                                                         | ~150 lines, reuses Daisy tokens; avoids FullCalendar's 100KB+ and its theming model.                                                                                                                                                   |
| **Out-of-territory warning**        | Red for `owned_by_other`, amber for `vacant`; both require an explicit confirm tick before save; stamped on `da_course_instances.out_of_territory_warning` + activity-logged.                                                                                                | PRD §6.2.                                                                                                                                                                                                                              |
| **Postmark wiring**                 | Still deferred to M3. M2 writes `da_email_sequences` rows `status='pending'`; nothing sends until M3's `send-emails` cron.                                                                                                                                                   | DECISIONS already defers Postmark; queue the data correctly now.                                                                                                                                                                       |
| **VAT / refund flow in M2**         | Not implemented. Refunds happen in the franchisee's Stripe dashboard; the portal shows status read-only.                                                                                                                                                                     | PRD §7.3 / out of M2 scope.                                                                                                                                                                                                            |
| **Mobile target (franchisee)**      | Phone-responsive with a collapsible bottom-nav on the franchisee shell only; HQ stays iPad-first.                                                                                                                                                                            | Franchisees use phones.                                                                                                                                                                                                                |

## Decisions taken in the M2 swarm execution plan (2026-05-27)

| Decision                   | Choice                                                                                                                                                                                                                                                                                         | Why                                                                                                                                      |
| -------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------- |
| **M2 migration numbering** | M2 migrations are **`019`+** (not the `014–016` the build plan originally named — those numbers are taken by M1's post-demo growth).                                                                                                                                                           | Audit found the repo at migration `018`.                                                                                                 |
| **RLS-audit migration**    | Near no-op. `da_ticket_types`, `da_discount_codes`, `da_private_clients` already carry `franchisee_own` policies; `da_course_instances` already has `stripe_payment_link`, `out_of_territory_warning`, `visibility`, `lat/lng`. Each new migration verifies the existing state before writing. | Schema was front-loaded in M1; don't re-add what exists.                                                                                 |
| **Parallel-agent process** | Per wave: 1 scaffold agent (sole owner of routes, shared types, migration numbers) → parallel build agents on disjoint file scopes → test-only peer-review agents (write new tests, never edit a sibling's source) → 1 verifier gate. Single wave branch off `m2/base`, no worktrees.          | Scaffold-first turns shared-file write-write conflicts into read-only dependencies; test-only reviewers preserve single-writer-per-file. |
